# Plan 45-01: Icon Resolver Fix

## Overview

`ClientPage.tsx` currently ignores the `icon` string stored in the DB and resolves icons by array index only. This plan adds a `getIcon()` resolver and a `FoodDrinkCombo` component to `ClientPage.tsx`, then wires both `resolvedFeatures` and `resolvedSteps` to use that resolver. It also expands the admin icon picker in `SettingsClient.tsx` with `Sandwich` and `CupSoda` entries so admins can set those icons from the UI.

Zero visual change is guaranteed because the DB currently stores the same icons the array-index fallback would have chosen. After this fix, admin-driven icon changes will propagate to the marketing page.

## Implementation Notes

- `FoodDrinkCombo` must be defined BEFORE `getIcon()` in source order, because `getIcon()` references it in the `iconMap` Record literal.
- Both `FoodDrinkCombo` and `getIcon()` must be defined BEFORE the `HowItWorks` and `FeatureBlocks` component functions that call `getIcon()`.
- The correct insertion point is after the `faqs` const (line 113) and before the `// ─── Components ─────` comment (line 115).
- The static `features` and `steps` arrays (lines 44-86) are NOT touched — they are the hardcoded fallback path when `data` is null/empty.
- No DB migrations, no API changes, no schema changes — this is a pure UI-layer fix.
