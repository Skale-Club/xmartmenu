Fix all 63 ESLint errors across the codebase. There are four error categories — apply the fix pattern consistently:

**1. `react-hooks/purity` — `Date.now` called during render (2 files)**

Files: `src/app/(admin)/dashboard/page.tsx` line 24, `src/app/(superadmin)/overview/page.tsx` line 10.

These are async Server Components so they run once on the server, but the lint rule still flags `Date.now()` during render. Fix: extract the date calculation BEFORE the component function body — move it to module scope or compute it in a `const` outside JSX. In a Server Component the cleanest fix is to declare the value at the top of the async function before any Supabase calls:

```ts
// dashboard/page.tsx — at top of the async function body (not in JSX)
const cutoff = new Date(Date.now() - 3 * 60 * 60 * 1000).toISOString().split('T')[0]
// then use cutoff in the Supabase query
```

Same pattern in `overview/page.tsx` — move `todayBRT` assignment to inside the async function but before any conditional or JSX. The rule requires the call to happen in an async context, not during render. Since these are async Server Components, declaring `const todayBRT = new Date(Date.now() - ...).toISOString().split('T')[0]` at the very top of the async function body (before await calls) satisfies the rule.

**2. `react-hooks/set-state-in-effect` — setState called synchronously inside useEffect (3 files)**

Files:
- `src/app/(admin)/menu/categories/CategoriesClient.tsx` line 51: `setCategories(initial)` inside `useEffect`
- `src/components/admin/CopyMenuUrl.tsx` line 12: `setUrl(...)` inside `useEffect`
- `src/components/menu/MenuPage.tsx` line 712: `setImageIndex(0)` inside `useEffect`

Pattern: the rule prohibits calling setState synchronously at the top level of a useEffect body without a conditional. These are synchronous derivations from props — the correct fix is to remove the useEffect and compute the value directly as the initial state or a derived value:

For `CategoriesClient.tsx`: `setCategories(initial)` with dep `[initial]` is a prop-sync pattern. Replace with: initialize state directly from prop — `const [categories, setCategories] = useState(initial)`. Remove the useEffect that sets categories. If the component needs to respond to `initial` changing from parent, use a `key` prop on the component at the call site (preferred), OR keep the useEffect but wrap the setState in a condition: `if (/* some guard */)`. The simplest correct fix is `useState(initial)` without the useEffect, since `initial` is the server-fetched initial value that won't change during the component's lifetime.

For `CopyMenuUrl.tsx`: `setUrl(window.location.origin + path)` in a useEffect is a client-side-only pattern (needs `window`). Fix: initialize state as `''` and keep the useEffect, but compute with a ref or move to an event handler. Actually the cleanest fix for this rule is: since `window.location.origin` is stable, initialize state with a lazy initializer: `useState(() => typeof window !== 'undefined' ? \`${window.location.origin}${path.startsWith('/') ? path : \`/${path}\`}\` : '')` and remove the useEffect entirely. This avoids setState in effect.

For `MenuPage.tsx` line 712: `setImageIndex(0)` resets the image carousel when `product.id` changes. Fix: this is a key-reset pattern. Pass a `key={product.id}` prop to the component that contains this useEffect so React re-mounts it when product changes, making the `setImageIndex(0)` unnecessary. If that component is defined inline in MenuPage.tsx, extract it or add `key={product.id}` at the parent call site. Alternatively, use a ref to track previous product.id and only call setState inside an `if (prevId !== product.id)` guard.

Pattern for each file:

```tsx
// Replace: <a href="/dashboard">Dashboard</a>
// With:    <Link href="/dashboard">Dashboard</Link>
```

**4. `@typescript-eslint/no-explicit-any` — explicit `any` types (many files)**

Files: `layout.tsx` (admin), `BrandingClient.tsx`, `public/[slug]/page.tsx`, `settings/SettingsClient.tsx`, `api/onboarding/route.ts`, `api/orders/route.ts`, `get-effective-tenant.ts`.

Create `.github/workflows/ci.yml` with exactly the spec from CONTEXT.md decisions section. The file content:

```yaml
name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  ci:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
      - run: npm ci
      - run: npm run lint
      - run: npm run build
    env:
      NEXT_PUBLIC_SUPABASE_URL: ${{ secrets.NEXT_PUBLIC_SUPABASE_URL }}
      NEXT_PUBLIC_SUPABASE_ANON_KEY: ${{ secrets.NEXT_PUBLIC_SUPABASE_ANON_KEY }}
      SUPABASE_SERVICE_ROLE_KEY: ${{ secrets.SUPABASE_SERVICE_ROLE_KEY }}
```

Key decisions (per CONTEXT.md):
- Node 20 matches `@types/node: ^20` in package.json
- `cache: 'npm'` uses setup-node's built-in npm cache — no separate cache step needed
- `npm ci` (not `npm install`) for reproducible installs in CI
- Lint runs BEFORE build — fail fast, lint errors are cheaper to surface
- `npm run build` (next build) runs TypeScript compilation — TypeScript errors fail the step (CI-02)
- Env vars needed: Next.js build accesses `NEXT_PUBLIC_SUPABASE_URL` and `NEXT_PUBLIC_SUPABASE_ANON_KEY` at build time for the `!` assertions in the Supabase client. `SUPABASE_SERVICE_ROLE_KEY` is needed for server-side references. All sourced from GitHub secrets.
- No `timeout-minutes` — default GitHub timeout (6h) is fine for this workflow
- No test step — no test suite exists yet (deferred per CONTEXT.md)

After creating the file, run `npm run build` locally to verify the build passes with current code. The build also runs TypeScript, so this validates CI-02. If the build fails, fix the errors before considering this task done.
