# Phase 10: Image Seeding — Context

**Gathered:** 2026-05-06
**Status:** Ready for planning

<domain>
## Phase Boundary

Superadmin-only AI image seeding for tenant menus. This phase adds a "Seed images" trigger in the superadmin tenant detail page that kicks off a GitHub Actions workflow — generating a cover photo and per-product images via Gemini, converting to WebP, and uploading to Supabase Storage. The UI polls for job completion. No Vercel function runs the generation itself.

Text seeding, DB schema, and AI infrastructure (ai_usage, sanitizeForPrompt, revalidatePath) are inherited from Phase 9 and are NOT modified here.

</domain>

<decisions>
## Implementation Decisions

### Who Uses AI (inherited)
- **D-01 (from Phase 9):** Image seeding is superadmin-only. Tenants never see generation buttons or loading states.

### Image Generation Model
- **D-02:** `gemini-3.1-flash-image-preview` for all image generation — cover photo and per-product images. Chosen for better quality, 512–4K resolution support, and thinking mode. Uses `@google/genai` SDK directly (NOT `@ai-sdk/google` which is text-only) — requires adding `@google/genai` as a new package dependency.

### Image Sources
- **D-03:** All images are AI-generated via Gemini. No stock photo services (no Pexels, no Unsplash). Every cover photo and every product image is generated by `gemini-3.1-flash-image-preview`.

### Cover Photo Style
- **D-04:** Cover photo prompt uses the tenant's `business_type` and `company_name` to generate a generic restaurant interior — warm, inviting, professional food/hospitality photography style. The real restaurant photo upload (existing feature) remains separate and takes priority if the tenant has already uploaded one; the AI-seeded cover only fills the gap when no cover exists.

### Per-Product Image Prompt
- **D-05:** Claude's discretion for prompt construction. Planner decides the exact prompt strategy — likely product name + category + business type, professional food photography style. Additive only: skip products that already have an `image_url`.

### Execution Environment — GitHub Actions
- **D-06:** Image generation runs in a GitHub Actions workflow, NOT in a Vercel serverless function. Vercel's function duration limit (60s Hobby / 300s Pro) is too short for generating cover + N product images in sequence. The Vercel API route only: (1) creates a job record in the DB, (2) dispatches the GH Actions workflow via GitHub API (`workflow_dispatch` or `repository_dispatch`), (3) returns the `job_id` immediately.

### Job Tracking & Polling
- **D-07:** A new `ai_jobs` table tracks image seeding jobs: `(id, tenant_id, feature_key, status, created_at, completed_at, error_message)`. Status values: `pending` → `running` → `complete` | `failed`. The GH Actions workflow updates this row as it progresses and writes final status on completion.
- **D-08:** The superadmin UI polls a Vercel status endpoint (e.g. `GET /api/superadmin/tenants/[id]/seed-status?jobId=...`) every ~3 seconds until status is `complete` or `failed`. On `complete`, the UI refreshes the tenant detail page to show the newly seeded images.

### Data Writing (inherited pattern)
- **D-09 (from Phase 9):** Additive only — skip products that already have an `image_url`. Skip cover if tenant already has a cover photo. Safe to run multiple times.
- **D-10 (from Phase 9):** `revalidatePath()` called after all images are uploaded so the public menu ISR cache is invalidated.

### Image Pipeline
- **D-11:** Gemini returns images as base64 `inline_data`. The GH Actions Node.js script decodes base64 → Buffer → Sharp WebP conversion → upload to Supabase Storage `tenant-assets` bucket. Follows the same bucket + naming convention as the existing upload route (`{tenantId}/cover.webp`, `{tenantId}/products/{productId}.webp`).
- **D-12:** The existing `validateAndConvertToWebP()` in `src/lib/upload.ts` takes a `File` object (browser API). The GH Actions script needs a Buffer-based variant — either a new utility or inline Sharp call in the workflow script.

### Infrastructure (inherited)
- **D-13 (from Phase 9):** `ai_usage` tracks image generation calls per tenant — `feature_key: 'image_seeding'`. Not a blocking gate; informational only.
- **D-14 (from Phase 9):** `sanitizeForPrompt()` applied to `company_name` and `business_type` before any prompt interpolation.

### Claude's Discretion
- Exact prompt text for cover photo and per-product images — planner decides based on best practices for food/restaurant photography prompts
- Whether to generate all products in parallel or sequentially within the GH Actions job (parallelism tradeoffs: speed vs Gemini rate limits)
- GH Actions trigger type: `workflow_dispatch` vs `repository_dispatch` — planner picks based on simpler auth model
- Exact polling interval and timeout in the UI (3–5s interval, ~5 min timeout is reasonable)
- Whether `ai_jobs` is a new migration (023) or piggybacks on `ai_usage` with a status column

</decisions>

<specifics>
## Specific Ideas

- Cover photo aspect ratio: `16:9` (banner format fits the tenant cover slot)
- Product photo aspect ratio: `1:1` (square thumbnails, consistent with product grid)
- GH Actions script runs in Node.js, installs `@google/genai` + `@supabase/supabase-js` + `sharp` as job dependencies (or from project `package.json` if checked out)
- The `ai_jobs` table row is created by the Vercel trigger route; GH Actions updates it to `running` on start, `complete` or `failed` on finish
- Polling stops on `complete` (refresh images) or `failed` (show error message to superadmin)
- `GOOGLE_GENERATIVE_AI_API_KEY` (already added in Phase 9) reused — same key works for image models
- `GITHUB_TOKEN` / `GH_PAT` needed in Vercel env vars to dispatch GH Actions workflow

</specifics>
