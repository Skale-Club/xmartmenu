Rationale for each key:
- `numberOfRuns: 1` — single run per URL keeps CI fast; production URL scores are stable (synthetic).
- `preset: "lighthouse:no-pwa"` — suppresses PWA category failures (this app has no service worker); focuses assertions on real categories.
- `categories:performance` — maps to the Lighthouse Performance score shown in PageSpeed Insights (0–1 scale; 0.88 = score 88).
- `aggregationMethod: "pessimistic"` — if numberOfRuns ever increases, takes the worst run to avoid masking regressions.
- `target: "temporary-public-storage"` — uploads report HTML to LHCI temporary storage (no LHCI server needed); link appears in GitHub Actions log.

Create `.github/workflows/lighthouse-ci.yml` with the following exact content:

```yaml
# Lighthouse CI — Performance regression gate
#
# Blocks merge when Lighthouse mobile Performance score drops below 0.88.
#
# Threshold rationale:
#   Phase 14 baseline:  / = 100 mobile,  /restaurante-teste = 94 mobile
#   Phase 16 target:    both routes >= 90 mobile
#   Phase 17 threshold: 0.88 (2-point buffer below target for variance tolerance)
#
# Audits production URLs — no local server needed.
# If production is down during CI, this job will fail (intended behavior:
# a down production site IS a regression that should block merge).

name: Lighthouse CI

on:
  pull_request:
    branches:
      - main

jobs:
  lighthouse:
    name: Lighthouse Performance Audit
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Run Lighthouse CI
        uses: treosh/lighthouse-ci-action@v12
        with:
          configPath: .lighthouserc.json
          uploadArtifacts: true
          temporaryPublicStorage: true
```

Important decisions embedded in this YAML:
- `on: pull_request` only (not push to main) — the gate is pre-merge, not post-merge. Push-to-main runs would add noise and re-audit an already-merged state.
- No `secrets` needed — auditing public production URLs requires no auth tokens.
- No `setup-node` step — `treosh/lighthouse-ci-action` ships its own Node environment.
- `uploadArtifacts: true` — attaches report HTML to the Actions run for debugging regressions.
- `temporaryPublicStorage: true` — matches the upload.target in lighthouserc.json; posts a public report link in the PR check.
- No `continue-on-error` — a failing performance score MUST block merge (that is the point of this phase).
