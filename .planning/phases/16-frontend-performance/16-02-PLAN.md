**ISR analysis:**

Current values:
- `src/app/(public)/[slug]/page.tsx` — `export const revalidate = 60`
- `src/app/(public)/[slug]/[menuSlug]/page.tsx` — `export const revalidate = 60`
- Landing `src/app/page.tsx` — `export const dynamic = 'force-static'` (do not touch)

**Assessment for revalidate = 60 on public menu routes:**
- Menu data (products, categories, prices) changes only when a tenant actively edits their menu in the admin dashboard — typically 0-10 times per day, not once per minute
- End-users (customers) scanning QR codes need current menu data — stale menus show wrong prices or unavailable items
- 60 seconds is already conservative (short enough that menu changes propagate within 1 minute)
- Going shorter (e.g., 30s) would increase Supabase query load with minimal benefit
- Going longer (e.g., 300s) risks stale data when a tenant urgently disables a sold-out item

**Decision:** Retain `revalidate = 60` on both public menu routes. No code change. Document justification.

**Verify landing page is force-static:**
```bash
grep "force-static\|revalidate" src/app/page.tsx
```
If `force-static` is present — confirmed, document it. If missing, add `export const dynamic = 'force-static'` to `src/app/page.tsx` (but per Phase 14 baseline landing is already optimal, so this is likely already present).
