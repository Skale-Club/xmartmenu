Run the bundle analyzer build from the project root:

```powershell
$env:ANALYZE = "true"
npm run build
```

After the build completes, read the generated chunk sizes. The analyzer writes HTML treemaps to `.next/analyze/`. To extract numeric sizes, inspect the static chunk files directly:

```powershell
# List all JS chunks in .next/static/chunks/ sorted by size descending
Get-ChildItem -Path ".next/static/chunks" -Filter "*.js" -Recurse |
  Sort-Object Length -Descending |
  Select-Object -First 20 Name, @{N='SizeKB';E={[math]::Round($_.Length/1KB, 1)}} |
  Format-Table -AutoSize
```

Also check `.next/static/chunks/pages/` and `.next/server/chunks/` if present.

From the output, identify the top 5 largest chunks by size. For each chunk, determine:
- Is it a node_modules vendor bundle? (filename contains "framework", "vendor", or a package name) → lazy-load candidate: NO (framework code, can't be lazy-loaded without breaking the app)
- Is it a route-specific page chunk? (filename starts with a route path or "app/") → lazy-load candidate: YES if the route is not on the critical render path
- Is it a shared component bundle? → lazy-load candidate: YES if the component is below the fold or behind a user interaction

Record the raw data as a list. This list will be written into 14-BASELINE.md in Plan 03.

Per D-03: Extract top 5 largest chunks (CONTEXT.md says top 5, despite ROADMAP saying top 3 — use top 5 for richer baseline).
