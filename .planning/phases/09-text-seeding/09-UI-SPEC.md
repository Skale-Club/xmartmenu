## Component Inventory

All components follow existing hand-built patterns. No new component primitives.

### 1. AI Tools Section Container

```
<section>  (placed below the Tabs block, inside the outer <div className="p-8 max-w-4xl">)
  <div className="mt-8 bg-white border border-zinc-200 rounded-xl p-5">
    <h2 class="text-sm font-semibold text-zinc-900 mb-1">AI Tools</h2>
    <p class="text-xs text-zinc-400 mb-4">{business_type context line}</p>
    {bulk seed buttons row}
    {status area}
  </div>
</section>
```

**Placement:** After the closing `</div>` of the Tabs section, before the outer container closes. Does NOT go inside any tab.

### 2. Business Type Context Line

Displays the tenant's business context so superadmin can confirm before seeding.

```
<p class="text-xs text-zinc-400 mb-4">
  Seeding for: <span class="font-medium text-zinc-700">{businessType ?? 'Unknown business type'}</span>
</p>
```

If `businessType` is null/empty, show an editable text input instead so superadmin can supply it before seeding. Input uses the existing `input` class pattern: `w-full px-3 py-2 border border-zinc-300 rounded-lg text-sm focus:outline-none focus:ring-2 focus:ring-zinc-900`.

### 3. Bulk Seed Buttons Row

Four buttons in a single `flex flex-wrap gap-2` row:

| Button | Label | Type variant |
|--------|-------|-------------|
| Seed everything | "Seed menu" | `bg-zinc-900 text-white` (primary / accent) |
| Seed categories only | "Seed categories" | `border border-zinc-200 text-zinc-700` (secondary) |
| Seed products only | "Seed products" | `border border-zinc-200 text-zinc-700` (secondary) |
| Seed copy only | "Seed copy" | `border border-zinc-200 text-zinc-700` (secondary) |

Primary button class: `bg-zinc-900 text-white px-4 py-2 rounded-lg text-sm font-medium hover:bg-zinc-800 disabled:opacity-50 transition-colors`

Secondary button class: `border border-zinc-200 text-zinc-700 bg-white px-4 py-2 rounded-lg text-sm font-medium hover:bg-zinc-50 disabled:opacity-50 transition-colors`

**All four buttons disabled during any active seed operation.** Prevents concurrent calls.

### 4. Per-Item "Seed" Inline Button

Placed in the Menus tab, inline with the "Add category" and "Add product" inputs. Appears as a secondary action alongside the existing submit button for each input row.

Button class: `border border-zinc-200 text-zinc-700 bg-white px-3 py-1 rounded-lg text-xs font-medium hover:bg-zinc-50 disabled:opacity-50 transition-colors`

Label: "Seed" (verb only — intentional: spatial context of the inline form row makes the noun implicit; adding "category" or "product" would overflow the compact row layout)

**Layout:** `flex gap-2 items-center` row containing the existing input + existing "Add" button + new "Seed" button. The "Seed" button is placed to the RIGHT of the "Add" button.

### 5. Loading State

**During bulk seed (2-20s):** Replace button label with inline text indicator. Do NOT use a spinner component (none exists in project).

Pattern (copied from existing `inviteLoading` pattern):
```
{seedLoading ? 'Seeding...' : 'Seed menu'}
```

Additionally show a status line below the buttons:
```
<p class="text-xs text-zinc-400 mt-3 animate-pulse">
  Generating menu content — this may take up to 20 seconds...
</p>
```

Use Tailwind's built-in `animate-pulse` class (already available in Tailwind v4).

**During per-item seed:** Replace "Seed" label with "Seeding..." and disable the button.

### 6. Success Feedback Banner

After seeding completes, show a green banner (same pattern as the credentials banner in `TenantDetailClient.tsx`):

```
<div class="bg-green-50 border border-green-200 rounded-xl p-4 mt-4">
  <div class="flex items-center justify-between">
    <p class="text-sm font-medium text-green-800">{success message}</p>
    <button class="text-green-500 hover:text-green-700 text-xl">✕</button>
  </div>
</div>
```

Banner is dismissible (✕ button sets state to null). Auto-dismiss: none — superadmin must dismiss manually.

### 7. Error State

Reuses existing error pattern from `TenantDetailClient.tsx`:

```
<div class="bg-red-50 border border-red-200 rounded-xl px-4 py-3 mt-4 text-sm text-red-700 flex items-center justify-between">
  {error message}
  <button class="ml-4 text-red-400 hover:text-red-600">✕</button>
</div>
```

## Interaction States

### State Machine: Bulk Seed Buttons

```
idle
  → [click any seed button]
seeding (all 4 buttons disabled, "Seeding..." label on active button, pulse message shown)
  → [API returns 200]
success (banner shown, all buttons re-enabled)
  → [dismiss banner OR click another seed]
idle
  → [API returns error]
error (error banner shown, all buttons re-enabled)
  → [dismiss error]
idle
```

### State Machine: Per-Item Seed Button

```
idle
  → [click "Seed"]
seeding (this "Seed" button shows "Seeding..." and is disabled; other buttons unaffected)
  → [API returns 200]
idle (no persistent banner — the item appears in the tenant's admin immediately)
  → [API returns error]
error (inline text "Failed — retry?" below the input row, auto-clears after 5s)
```

### Menu selector requirement

The seed API requires a `menuId`. If the tenant has multiple menus, a `<select>` dropdown must appear above the bulk buttons to choose which menu to seed. If only one menu exists, auto-select it silently and do not show the selector.

```
<select class="px-3 py-2 border border-zinc-300 rounded-lg text-sm focus:outline-none focus:ring-2 focus:ring-zinc-900 mb-4">
  <option value="">Select menu to seed...</option>
  {menus.map(m => <option value={m.id}>{m.name}</option>)}
</select>
```

All seed buttons are disabled until a menu is selected.

**Note on per-item buttons:** The existing Menus tab only shows a menus list table — it has no "Add category" or "Add product" inputs. These forms are in the **tenant admin UI**, not the superadmin panel. Re-reading CONTEXT.md D-03 and RESEARCH.md "What must change in TenantDetailClient.tsx" point 3: per-item Seed buttons go "next to the Add category and Add product inputs in the superadmin tenant view." Since those inputs do NOT currently exist in the superadmin view, the planner must decide whether to add lightweight Add Category / Add Product forms to the Menus tab OR interpret "per-item" as a separate inline section within the AI Tools card. The UI contract defers this to the planner but specifies the button appearance and placement contract above.

## Implementation Notes for Executor

1. State variables to add to `TenantDetailClient.tsx`:
   - `seedLoading: boolean` — true during any seed operation
   - `seedStatus: { type: 'success' | 'error'; message: string } | null` — banner content
   - `selectedMenuId: string` — menu picker value (default to first menu if only one)
   - `businessTypeInput: string` — editable field if `businessType` prop is null

2. Prop additions to `TenantDetailClient`:
   - `businessType: string | null` — from `tenant_settings.business_type` (or fallback field; see RESEARCH.md Pitfall 2)
   - `supportedLanguages: string[]` — from `menus.supported_languages` for the selected menu
   - `menus` already passed as `initialMenus` — no change needed

3. `page.tsx` must add to its selects:
   - `tenant_settings`: add `business_type` if column exists (verify in migration 007)
   - `menus`: add `supported_languages` to the select string

4. The `animate-pulse` class is used only on the loading message paragraph, NOT on buttons (buttons just go `opacity-50` via `disabled:opacity-50`).

5. Typography: weight 500 (`font-medium`) appears in existing button class patterns already in the codebase. It is NOT a declared design token. Do not introduce new `font-medium` usage beyond what button classes already carry from the existing pattern library.

6. API call pattern (mirrors existing `handleInvite` pattern):
   ```typescript
   async function handleSeed(type: string) {
     setSeedLoading(true)
     setSeedStatus(null)
     const res = await fetch(`/api/superadmin/tenants/${tenant.id}/seed`, {
       method: 'POST',
       headers: { 'Content-Type': 'application/json' },
       body: JSON.stringify({ type, menuId: selectedMenuId }),
     })
     const data = await res.json()
     if (!res.ok) {
       setSeedStatus({ type: 'error', message: data.error ?? 'Seeding failed. Check the API logs and retry.' })
     } else {
       setSeedStatus({ type: 'success', message: buildSuccessMessage(type, data) })
     }
     setSeedLoading(false)
   }
   ```
