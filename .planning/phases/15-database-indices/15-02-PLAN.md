1. SELECT id, is_active, tenant_settings(orders_enabled)
   FROM tenants

   -- Lookup by PK (id); PK always has implicit B-tree index — should be Index Scan

2. INSERT INTO orders (tenant_id, customer_name, customer_phone, status, total) VALUES (...)
   -- No lookup; pure insert. No index needed for performance.

3. INSERT INTO order_items (order_id, product_id, product_name, quantity, unit_price, notes, selected_options) VALUES (...)
   -- No lookup; pure insert. No index needed for performance.

GET /api/orders — 1 query:
4. SELECT *, order_items(*)
   FROM orders
   WHERE tenant_id = $tenantId
   ORDER BY created_at DESC
   -- idx_orders_tenant ON orders(tenant_id) EXISTS
   -- idx_orders_created_at ON orders(created_at) EXISTS
   -- Check if both are used, or only one

Auth middleware / RLS helper queries (run implicitly per Supabase RLS policies):
5. SELECT tenant_id FROM profiles WHERE id = auth.uid()         -- auth_tenant_id() helper
6. SELECT EXISTS(SELECT 1 FROM profiles WHERE id = auth.uid() AND role = 'superadmin')  -- is_superadmin() helper

Note: auth.uid() is resolved internally by Supabase auth. For EXPLAIN ANALYZE we simulate with a known UUID.

No profiles(id) index needed — id is the PK.
No profiles(role) index: the function filters on id (PK) first, so role filter is on a single row.
