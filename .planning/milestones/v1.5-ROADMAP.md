# Milestone v1.5: Image Optimization

**Status:** ✅ SHIPPED 2026-05-08
**Phases:** 18-20
**Total Plans:** 7

## Overview

End-to-end image optimization for xmartmenu. Fixed the upload pipeline to enforce WebP conversion on all paths, migrated all admin and superadmin image rendering to `next/image`, and introduced a storage abstraction layer (`IStorageClient`) that makes migrating from Supabase Storage to Hetzner Object Storage (or any S3-compatible provider) a single env-var change.

Also applied DB migration 024 — 4 missing performance indices on the public menu query path (menus, categories, products tables), eliminating sequential scans on every public page load.

## Phases

### Phase 18: WebP Upload Pipeline

**Goal:** Enforce WebP conversion on every upload path — no raw PNG/JPEG reaches Supabase Storage.
**Plans:** 3 plans (executed inline in 18-01)

Plans:
- [x] 18-01: Wire `validateAndConvertToWebP` into ProductsClient upload (new `/api/admin/products/upload` server route); fix superadmin upload/route.ts; add `formats: ['image/webp']` + `deviceSizes` to next.config.ts

**Decisions:**
- Admin product upload moved to server-side API route — `sharp` cannot run in browser/client components
- Used `new Uint8Array(buffer)` wrapper to satisfy TypeScript's `BlobPart` type constraint

### Phase 19: Admin next/image Migration

**Goal:** Replace all raw `<img>` tags in admin/superadmin with `next/image` for LCP and lazy-load gains.
**Plans:** 3 plans

Plans:
- [x] 19-01: `BrandingClient.tsx` — logo (80×80 fixed) + banner (`fill`, priority)
- [x] 19-02: `ProductsClient.tsx` — admin grid thumbnails (56×56) + upload preview (64×64)
- [x] 19-03: `TenantsClient.tsx` (40×40) + `TenantDetailClient.tsx` (40×40)

**Decisions:**
- Remove Tailwind `w-N h-N` classes when `next/image` takes over sizing — avoids conflicting dimension hints
- `next/image` requires explicit `sizes` prop per layout context — set correctly per component

### Phase 20: Storage Abstraction

**Goal:** Decouple all server-side storage calls from Supabase SDK behind `IStorageClient`.
**Plans:** 1 plan

Plans:
- [x] 20-01: `src/lib/storage/index.ts` — `IStorageClient` interface + `SupabaseStorageClient` + `S3StorageClient` + `getStorageClient()` factory

**Files refactored:** 5 routes migrated from direct supabase.storage to `getStorageClient()`
**Dependencies added:** `@aws-sdk/client-s3`, `@aws-sdk/s3-request-presigner` (lazy imports)

## DB Migration 024 (applied 2026-05-08)

4 missing indices for the public menu path:

```sql
CREATE INDEX IF NOT EXISTS idx_menus_tenant    ON menus(tenant_id);
CREATE INDEX IF NOT EXISTS idx_menus_slug      ON menus(slug);
CREATE INDEX IF NOT EXISTS idx_categories_menu ON categories(menu_id);
CREATE INDEX IF NOT EXISTS idx_products_menu   ON products(menu_id);
```

## Milestone Summary

**Key Decisions:**
- `validateAndConvertToWebP` moved behind server-side API route (sharp = Node.js only)
- `IStorageClient` factory pattern — provider swap is env-var only (`STORAGE_PROVIDER=s3`)
- `S3StorageClient` uses lazy `@aws-sdk` imports to prevent client-bundle pollution
- UNIQUE(tenant_id, slug) composite index does NOT serve single-column filters — separate indices required

**Issues Resolved:**
- Admin product upload was bypassing WebP converter (raw PNG/JPEG reaching storage)
- Superadmin upload route also writing raw bytes
- 4 sequential scans per public page load eliminated by migration 024

**Deferred (Out of Scope):**
- Responsive variants (256w/512w/1024w)
- AVIF format
- CDN Cache-Control headers
- Backfill of legacy storage
